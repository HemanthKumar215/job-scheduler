2. System Architecture

2.1 Overview

Aether Scheduler is a production-grade, multi-tenant distributed job scheduling engine built as a monorepo of three independently deployable services (API, Worker, Frontend) backed by PostgreSQL and Redis. It handles high-concurrency job submission, at-least-once execution, configurable retry backoff, dead-letter queuing, and live WebSocket status updates.

2.4 Job State Machine

[API POST] -> QUEUED -> CLAIMED -> RUNNING -> COMPLETED (terminal, success) ^ | | | attempts < maxRetries | FAILED (auto-retry: scheduledAt = future) | | +------ [manual retry] -----+ | attempts >= maxRetries DLQ (terminal, quarantined) | [manual requeue-dlq] -> QUEUED

3. Database Design & ER Diagram

3.1 Design Principles

  • Fully Normalized (3NF): No data duplication; all entities in Third Normal Form
  • Cascade-aware deletes: Org deletion cascades through projects, queues, jobs. High-volume tables (JobLog) use soft string references to avoid lock contention
  • UUID primary keys: Prevents ID enumeration; enables future horizontal sharding
  • Native enum types: JobStatus, OrgRole, RetryStrategy enforced at PostgreSQL level
  • Targeted composite indexes: 3 indexes designed for hot-path queries with verified EXPLAIN ANALYZE plans

3.2 Entity-Relationship Diagram

User |--o{ OrganizationMember }o--| Organization | 1..* RetryPolicy ||--o{ Queue }o--|| Project | 0..* User ||--o{ Job }o--|| Queue | | 1..* 0..* JobExecution DeadLetterQueue Worker ||--o{ WorkerHeartbeat Job --o{ JobLog (soft ref, no FK constraint) Relationships: User has many OrganizationMember, Job (assigned) Organization has many OrganizationMember, Project Project has many Queue, Job Queue has many Job (via RetryPolicy) Job has many JobExecution, DeadLetterQueue, JobLog

3.4 Job Model Schema (Core Entity)

model Job {
  id             String         @id @default(uuid())
  payload        Json
  status         JobStatus      @default(QUEUED)
  priority       Int            @default(1)
  projectId      String
  queueId        String
  userId         String?        // Optional user assignment
  scheduledAt    DateTime       @default(now())
  cronExpression String?
  attemptCount   Int            @default(0)
  idempotencyKey String?
  batchId        String?
  correlationId  String

  project     Project          @relation(...)
  queue       Queue            @relation(...)
  user        User?            @relation(..., onDelete: SetNull)
  executions  JobExecution[]
  dlqEntries  DeadLetterQueue[]

  @@unique([projectId, idempotencyKey])
  @@index([status, scheduledAt, priority], name: "idx_job_poll")
  @@index([status, projectId], name: "idx_job_status_project")
}

5. Design Decisions & Major Trade-offs

5.1 Claiming Strategy: SELECT FOR UPDATE SKIP LOCKED

ApproachProsCons
Selected: PostgreSQL SKIP LOCKEDZero extra dependency, transactional consistency, no split-brain, debuggableTied to PostgreSQL
Redis distributed lockFast, language-agnosticSplit-brain risk if DB write fails; Redlock complexity
Dedicated broker (RabbitMQ/SQS)Mature, highly scalableExtra infra, ops complexity
DecisionSKIP LOCKED keeps the entire claim-execute-update cycle in PostgreSQL transactions, correct-by-construction without distributed coordination.

5.2 At-Least-Once Execution

True exactly-once is impossible: a worker crashing after completing an action but before writing COMPLETED causes the job to be re-queued by the heartbeat supervisor.

Mitigations: Idempotency-Key header (dedup at submission), correlationId for tracing, consumer guidance to write idempotent handlers.

5.3 Database Index EXPLAIN ANALYZE Results

Plan: Bitmap Index Scan on idx_dlq_job
  Planning Time: 0.233 ms | Execution Time: 0.029 ms

SELECT * FROM "WorkerHeartbeat" WHERE "workerId" = 'w' ORDER BY "timestamp" DESC LIMIT 5;

Plan: Bitmap Index Scan on idx_worker_heartbeat
  Planning Time: 0.178 ms | Execution Time: 0.022 ms

5.4 Heartbeat-Based Failure Detection

Workers write WorkerHeartbeat every 10s. Supervisor queries workers where lastHeartbeatAt < NOW()-30s, marks them DEAD, and re-queues their orphaned CLAIMED/RUNNING jobs back to QUEUED.

No lost jobsJobs only leave QUEUED inside a DB transaction; if it fails, job stays QUEUED
No double-executionSKIP LOCKED ensures exactly one worker claims each row

9.2 Critical CI Gotchas

Gotcha 1: Never use prisma migrate dev in CI migrate dev is interactive and can auto-generate migrations from schema drift. Always use: npx prisma migrate deploy --schema=prisma/schema.prisma
Gotcha 2: Do not quote .env values DATABASE_URL="postgresql://..." with literal quotes causes Prisma to receive an invalid URL. Correct: DATABASE_URL=postgresql://postgres:password@localhost:5432/db Wrong: DATABASE_URL="postgresql://postgres:password@localhost:5432/db"
Gotcha 3: .env discovery in monorepos Prisma CLI uses cwd; PrismaClient uses the prisma/ subdirectory. Use workflow-level env: in GitHub Actions so all processes get DATABASE_URL via process.env.
